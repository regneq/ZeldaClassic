    I read this article by Troy Gilbert: troygilbert.com/deconstructing-zelda/movement-mechanics/

    In summary, he says that Link moves on a half grid (8x8). But, he also moves pixel by pixel. For example, if the player presses right, Link moves 1 pixel instead of moving all the way to the next half tile.

    To keep you aligned to the grid (to avoid getting snagged on corners, to be able to line up attacks with enemies, etc.), there is course correction. My interpretation was that, when not aligned to a half tile, and you're moving, you auto-move towards the nearest half tile. Here's a bit of what I have so far:
    Code:
    if (facing == NORTH || facing == SOUTH)
{
    // The center of the tile the player is on
    var player_tile = (x div TILE_SIZE) * TILE_SIZE + HALF_TILE;
       
    // If the player is aligned, we don't need to correct
    var aligned = (x == player_tile);
    if (!aligned)
    {
        // So I know to change his x by "dx" when he moves up/down
        correct_x = true;
           
        // Store the positions of the three possible tiles he could course correct to  
        var middle_tile, right_tile, left_tile;
       
        middle_tile = player_tile;
        right_tile = player_tile + TILE_SIZE;
        left_tile = player_tile - TILE_SIZE;

        // Pick the shortest
        var dist_to_middle_tile, dist_to_right_tile, dist_to_left_tile;
           
        dist_to_middle_tile = point_distance(x, 0, middle_tile, 0);
        dist_to_right_tile = point_distance(x, 0, right_tile, 0);
        dist_to_left_tile = point_distance(x, 0, left_tile, 0);
           
        if (min(dist_to_middle_tile, dist_to_right_tile, dist_to_left_tile) == dist_to_middle_tile)
        {
            targ_tile = middle_tile;    
        }
        else if (min(dist_to_middle_tile, dist_to_right_tile, dist_to_left_tile) == dist_to_right_tile)
        {
            targ_tile = right_tile;    
        }
        else if (min(dist_to_middle_tile, dist_to_right_tile, dist_to_left_tile) == dist_to_left_tile)
        {
            targ_tile = left_tile;    
        }
       
        // What x direction to move in when course correcting
        dx = (x < targ_tile ? 1 : -1);
    }
}
    After testing this out, it does work the way I intended it to, but it doesn't work the way Link moves in the Legend of Zelda. Clearly I am misunderstanding the article.

    I know this question is vague, so I apologize. I was wondering if anyone knows anything about this type of movement? Just anything that would set me off in the right direction. Thanks for reading.

    @Bentley - It's not clear from the article what method is used to align Link. But from the description it appears he is always pushed right or down to the next grid aligned coordinate. Therefore, you would decide which axis Link is moving on, horizontal or vertical, and then add 1 to x or y if that coordinate is not currently grid aligned. Something like:
    Code:
    var _xoffset = ( x mod 8 > 0 ? 1 : 0 );
    If you have GM2, as GM1.4 it looks like:
    Code:
    var _xoffset = 0;

if x mod 8 > 0 { _xoffset = 1 }
    Obviously you'd do the same thing for the y coordinate, and you'd have to decide which one based on which direction Link is moving but I believe you'll understand the jist.

    @Bentley - I'm going to take a guess that your origin X and Y of your link sprite is 8, 8, as opposed to 0, 0. In the case of the NES game, the origin of Link, as far as I know, is 0, 0. The reason this is important is because it informs which method we should use to align him. In the case of aligning him to a _full_ tile, we should be using mod 16, and > 8 instead, that way we push him an entire tile out, instead of part of one.

    To answer your question a bit better, the only way we could decide which course correction, left or right, is best would be to actually check the path Link is on and try to find the right course. However, this may or may not feel a bit janky to the player. Perhaps they wanted to run into the tree instead, and we are pushing them away from it! The benefit of the original NES version is the behavior is 100% consistent and the player learns to adapt their movement accordingly. It would be very easy to code a system like this where the player feels like they are fighting it constantly as it makes decisions about where it thinks they want to go. The purpose of including is because Link will get stuck on level geometry, and it allows him to deal with it. Consider the bushes in the lower right. If we use the basic 'push left/down' behavior, the player will be pushed into the bushes, therefore they will try to walk above the bushes before pressing right again. If they do it from the upper-right bushes, they'll instead be pushed down, and past the bush, if they press 'right' before being grid aligned. In either case, Link is moved into a consistent position, but more importantly doesn't have to precisely find the right y coordinate to pass by the bushes, the player can approximate.

    What I'm getting at is the system isn't necessarily made to cause Link to avoid obstacles, so much as it deals with obstacles he runs into by making his behavior more consistent. Or to put it another way, it's simply meant to make movement more fluid, not prevent collisions. However, all of that said and based on what you are asking I do believe my initial comment should be correct. Instead of aligning him to the nearest 8, you need to align him to the nearest 16.

    Code:
    var input_x, input_y;
input_x = key_right - key_left;
input_y = key_down - key_up;

x += input_x * move_speed;
y += input_y * move_speed;
 
var correct_x = (input_y != 0 && x mod TILE_SIZE != 0)
if (correct_x)
{
    x += (x mod TILE_SIZE > HALF_TILE ? 1 : -1);  // This one line saves a hundred lines and a bunch of needless variables
}
